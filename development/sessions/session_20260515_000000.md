**The discovery, in brief:** the previous session left a clean working tree at `f42ac95`. Picked up that work, ran tests against the changes, and committed it straight to dev (no feature branch — owner directive given the transfer urgency).

**What the change does:**

#### Schema — migration `2026_05_13_000001_add_refund_breakdown_to_readings.php`

Adds three columns to both `zreadings` and `shift_readings`:

```
vatable_on_refunds       double(15,2) | decimal(12,2)   default 0
vat_exempt_on_refunds    double(15,2) | decimal(12,2)   default 0
zero_rated_on_refunds    double(15,2) | decimal(12,2)   default 0
```

Previously only the rolled-up `vat_on_refunds` was tracked on these tables. The full BIR break-out (vatable / exempt / zero-rated) on refund rows was being lost, so reconciliation against the BIR Sales Book was incomplete.

#### `SaleController::refundReceipt` — real accounting accuracy fix

When a refund Sale row was being created, the BIR fields were being copied verbatim from the original sale:

```php
// BEFORE (wrong)
'vatable'    => $sale->vatable,
'vat'        => $sale->vat,
'vat_exempt' => $sale->vat_exempt,
'zero_rated' => $sale->zero_rated,
'sc_discount' => $sale->sc_discount,
'pwd_discount' => $sale->pwd_discount,
'sp_discount' => $sale->sp_discount,
'naac_discount' => $sale->naac_discount,
'vat_special_discounts' => $sale->vat_special_discounts,
```

Partial refund problem: if 3 of 10 items were refunded, the refund record would claim the VAT amount for all 10. Wrong amount on BIR forms.

Now it accumulates per refund line scaled by qty:

```php
foreach ($refundLine as $line) {
    $saleLine = SaleLine::with(...)->first();
    $refundVatable  += $line['qty'] * ($saleLine->vatable  ?? 0);
    $refundVat      += $line['qty'] * ($saleLine->vat      ?? 0);
    $refundVatExempt += $line['qty'] * ($saleLine->exempt   ?? 0);
    $refundZeroRated += $line['qty'] * ($saleLine->zero_rated ?? 0);
    $refundScDiscount  += $line['qty'] * ($saleLine->sc_discount  ?? 0);
    $refundPwdDiscount += $line['qty'] * ($saleLine->pwd_discount ?? 0);
    $refundSpDiscount  += $line['qty'] * ($saleLine->sp_discount  ?? 0);
    $refundNaacDiscount += $line['qty'] * ($saleLine->naac_discount ?? 0);
    $refundVatSpecialDiscounts += $line['qty'] * ($saleLine->vat_special_discounts ?? 0);
}
// then store rounded(2) on the refund Sale row
```

#### `XreadingController::apexReading` — query restructure

The aggregation was computing `vatable = sum(if(sale_type=0, vatable, -vatable))` — subtracting refunds from the sales total. Now that refunds get their own bucket:

```php
// BEFORE (refunds netted into main totals)
'ROUND(sum(if(sale_type = 0, vatable, -vatable)), 2) as vatable'

// AFTER (sales-only on main; refunds in dedicated columns)
'ROUND(sum(if(sale_type = 0, vatable, 0)), 2) as vatable',
'ROUND(sum(if(type = 1, vatable, 0)), 2) as vatable_on_refunds'
```

Same restructure for `vat`, `vat_exempt`, `zero_rated`. The main columns are sales-only; refunds live in their own break-out columns. Net is computed consumer-side: `vatable - vatable_on_refunds`. This makes BIR sales-book mapping reconcile sales vs. refunds independently.

## Deploy considerations for this change

The migration adds three columns each to `zreadings` and `shift_readings`. Both are additive (NOT NULL with default 0), so no data backfill needed.

```bash
# On prod (or staging):
php artisan migrate --force
```

**No client-app changes required.** The new columns are optional on the form requests (nullable / numeric). Existing apex_pos / apex_dashboard payloads continue to work as before — they just won't populate the new columns, which then default to 0. The Z-reading aggregation query writes the new columns automatically when a Z-reading is generated, so post-deploy Z-readings will have the breakdown populated.

**For BIR reporting code that reads these tables:** the meaning of `vatable`, `vat`, `vat_exempt`, `zero_rated` on `zreadings` has changed. They used to be net (sales minus refunds). Now they're sales-only. Any code that downstream-consumes these fields expecting net needs to be updated to compute `field - field_on_refunds` explicitly. **Search the codebase for read-paths of these columns** before downstream reporting work picks them up — likely admin Z-reading display views and any BIR report generation.
